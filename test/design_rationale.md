# 香港小姐竞选投票系统 - 设计思路和考量

## 一、数据库设计思路

### 1. 核心表结构设计

#### 1.1 voters（选民表）

**设计考量：**
- **手机号作为唯一标识**：香港地区手机号格式统一（8位数字），便于用户记忆和登录
- **香港身份证哈希存储**：出于隐私保护考虑，不存储明文身份证号，而是存储 SHA256 哈希值
- **验证码机制**：支持短信验证码验证，确保手机号真实性
- **登录安全**：实现登录失败次数限制和账户锁定机制，防止暴力破解
- **令牌认证**：使用 `has_secure_token` 生成安全的认证令牌，支持令牌过期刷新

**索引设计：**
- `phone` 唯一索引：确保一个手机号只能注册一次
- `hkid_hash` 唯一索引：确保一人一票（基于香港身份证）
- `auth_token` 索引：加速会话验证查询
- `phone_verified` 索引：便于统计已验证用户

#### 1.2 contest_sessions（赛事届次表）

**设计考量：**
- **多届赛事支持**：系统支持多届香港小姐竞选，每届独立管理
- **时间窗口管理**：分别设置报名和投票的开始/结束时间
- **状态自动计算**：根据当前时间自动判断赛事状态（未开始/进行中/已结束）
- **当前届次标记**：使用 `is_current` 标记当前活跃的届次

**索引设计：**
- `year` 唯一索引：确保每届赛事年份唯一
- `status` 索引：便于按状态筛选赛事
- `is_current` 索引：快速查询当前届次
- `voting_start_at` + `voting_end_at` 复合索引：优化投票时间窗口查询

#### 1.3 candidates（候选人表）

**设计考量：**
- **状态管理**：使用枚举管理候选人生命周期（待审核 -> 已通过 -> 已淘汰）
- **参赛编号**：为每位候选人分配唯一编号，便于展示和识别
- **软删除支持**：使用 `deleted_at` 实现软删除，保留历史数据
- **展示顺序**：支持自定义展示顺序，便于前端排序展示

**索引设计：**
- `candidate_number` 唯一索引：确保参赛编号唯一
- `status` 索引：筛选可投票的候选人
- `display_order` 索引：优化列表排序
- `deleted_at` 索引：排除已删除记录
- `contest_session_id` + `status` + `display_order` 复合索引：优化候选人列表查询

#### 1.4 votes（投票记录表）

**设计考量：**
- **一人一票约束**：使用数据库唯一索引确保每个选民只能投一票
- **软删除支持**：允许撤销投票，保留审计记录
- **环境信息记录**：记录 IP 地址和 User-Agent，用于防作弊审计
- **外键约束**：确保数据完整性，防止孤立记录

**索引设计：**
- `voter_id` 唯一索引：确保一人一票
- `candidate_id` 索引：统计候选人票数
- `candidate_id` + `created_at` 复合索引：查询投票时间分布
- `deleted_at` 索引：筛选有效投票
- `ip_address` 索引：检测异常投票行为

#### 1.5 candidate_photos（候选人照片表）

**设计考量：**
- **多照片支持**：一个候选人可以有多张照片（主照片 + 相册）
- **主照片标记**：使用 `is_primary` 标记主展示照片
- **元数据记录**：记录文件大小、尺寸等信息，便于前端优化
- **文件路径存储**：支持多种存储方式（本地、云存储）

**索引设计：**
- `candidate_id` 索引：查询候选人的照片
- `photo_type` 索引：按类型筛选照片
- `candidate_id` + `is_primary` 复合索引：快速查询主照片
- `candidate_id` + `display_order` 复合索引：照片排序

### 2. 数据类型选择

| 字段 | 类型 | 考量 |
|------|------|------|
| 手机号 | string | 香港手机号8位，使用字符串避免前导零丢失 |
| 身份证哈希 | string | SHA256 哈希值为64位十六进制字符串 |
| 状态 | string | 使用字符串枚举，便于扩展和阅读 |
| 时间戳 | datetime | 统一使用 UTC 时间，应用层转换时区 |
| 金额/数量 | integer | 票数等使用整数，避免浮点精度问题 |
| 布尔值 | boolean | 明确使用布尔类型，避免三态问题 |

### 3. NULL 属性设计

- **必须字段**：设置 `null: false` 确保数据完整性（如手机号、姓名等）
- **可选字段**：允许 NULL 表示该信息尚未提供（如 `approved_at` 表示尚未审核）
- **软删除字段**：`deleted_at` 默认为 NULL，删除时设置时间戳

## 二、REST API 设计思路

### 1. RESTful 设计原则

#### 1.1 资源命名

- 使用名词复数形式表示资源集合：`/candidates`、`/votes`
- 使用嵌套路径表示关联关系：`/candidates/:id` 表示单个候选人
- 使用动作路径表示非 CRUD 操作：`/voters/verification_code`

#### 1.2 HTTP 方法使用

| 方法 | 用途 | 示例 |
|------|------|------|
| GET | 获取资源 | `GET /candidates` 获取候选人列表 |
| POST | 创建资源 | `POST /votes` 提交投票 |
| DELETE | 删除资源 | `DELETE /votes` 撤销投票 |

#### 1.3 状态码使用

| 状态码 | 使用场景 |
|--------|----------|
| 200 OK | 请求成功，返回数据 |
| 201 Created | 资源创建成功（注册、投票） |
| 400 Bad Request | 请求参数格式错误 |
| 401 Unauthorized | 未提供认证信息或认证失败 |
| 404 Not Found | 资源不存在 |
| 422 Unprocessable Entity | 业务逻辑验证失败（已投票、验证码错误等） |
| 429 Too Many Requests | 请求过于频繁（限流、账户锁定） |
| 500 Internal Server Error | 服务器内部错误 |

### 2. API 设计考量

#### 2.1 注册流程

**发送验证码**：`POST /voters/verification_code`
- 验证香港身份证格式
- 生成6位数字验证码
- 发送短信（开发环境打印日志）
- 设置10分钟有效期

**用户注册**：`POST /voters`
- 验证验证码
- 创建或更新选民记录
- 存储身份证哈希值
- 返回用户信息

#### 2.2 登录流程

**用户登录**：`POST /sessions`
- 验证手机号和身份证哈希
- 实现登录失败次数限制（5次后锁定30分钟）
- 生成认证令牌
- 返回用户信息和令牌

**令牌刷新**：`POST /sessions/refresh`
- 验证当前令牌
- 生成新令牌
- 延长有效期（30天）

#### 2.3 候选人查询

**候选人列表**：`GET /candidates?page=1&per_page=20`
- 支持分页（默认20条/页）
- 只返回已审核通过的候选人
- 按展示顺序排序
- 包含票数统计和照片URL

**候选人详情**：`GET /candidates/:id`
- 返回候选人完整信息
- 包含赛事信息
- 包含投票状态（是否可投票）

#### 2.4 投票流程

**提交投票**：`POST /votes`
- 验证用户是否已登录
- 验证是否已投票（一人一票）
- 验证投票时间窗口
- 创建投票记录
- 更新候选人票数缓存

**撤销投票**：`DELETE /votes`
- 验证用户是否有投票
- 验证是否在撤销时限内（24小时）
- 软删除投票记录
- 更新候选人票数缓存

**我的投票**：`GET /votes/my_vote`
- 返回当前用户的投票记录
- 用于前端显示投票状态

### 3. 响应格式设计

#### 3.1 统一响应结构

```json
{
  "success": true/false,
  "message": "操作结果描述",
  "data": { ... },
  "errors": { ... }
}
```

**设计考量：**
- `success` 字段：明确标识操作是否成功
- `message` 字段：用户友好的提示信息
- `data` 字段：包含返回的业务数据
- `errors` 字段：详细的字段级错误信息

#### 3.2 分页响应

```json
{
  "success": true,
  "data": {
    "candidates": [...],
    "pagination": {
      "current_page": 1,
      "per_page": 20,
      "total_pages": 5,
      "total_count": 100
    }
  }
}
```

### 4. 安全设计

#### 4.1 认证机制

- **Bearer Token**：使用 HTTP Authorization 头传递令牌
- **令牌有效期**：30天，支持刷新
- **登录保护**：失败5次后锁定账户30分钟

#### 4.2 防作弊措施

- **IP 限流**：使用 Rack::Attack 限制请求频率
- **验证码**：短信验证码防止机器注册
- **一人一票**：数据库唯一索引约束
- **投票审计**：记录 IP 地址和 User-Agent

#### 4.3 数据保护

- **身份证哈希**：不存储明文身份证号
- **HTTPS**：生产环境强制使用 HTTPS
- **CORS**：限制跨域访问

## 三、性能优化考量

### 1. 数据库优化

- **索引优化**：为常用查询字段创建索引
- **复合索引**：优化多条件查询
- **外键约束**：确保数据完整性
- **软删除**：使用 `deleted_at` 而非物理删除

### 2. 缓存策略

- **票数缓存**：使用 Redis 缓存候选人票数，5分钟过期
- **缓存失效**：投票时主动刷新缓存
- **查询缓存**：利用 ActiveRecord 的查询缓存

### 3. 分页优化

- **数据库分页**：使用 `LIMIT` 和 `OFFSET` 实现
- **避免深分页**：限制最大页码，避免性能问题

## 四、扩展性考量

### 1. 多届赛事支持

- 使用 `contest_sessions` 表管理多届赛事
- 候选人关联到具体届次
- 支持历史数据查询

### 2. 多照片支持

- 独立的 `candidate_photos` 表
- 支持主照片和相册照片
- 可扩展支持视频等多媒体

### 3. 投票规则扩展

- `max_votes_per_voter` 字段支持多票制
- 可为不同届次设置不同规则
- 未来可扩展支持排名投票等复杂规则

---

## 五、总结

本设计充分考虑了香港小姐竞选投票系统的业务需求，在保证功能完整性的同时，注重：

1. **数据安全**：身份证哈希存储、软删除、审计日志
2. **系统安全**：验证码、限流、登录保护、一人一票约束
3. **性能优化**：索引优化、缓存策略、分页查询
4. **可扩展性**：多届赛事、多照片、灵活的投票规则
5. **代码质量**：清晰的注释、规范的命名、合理的分层

所有设计都遵循 Rails "约定大于配置" 的原则，确保代码的可维护性和可测试性。
